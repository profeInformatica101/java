Resumen JUnit

Configuración: dependencias de MAVEN, Anotaciones, Aserciones y Ejemplos

Si estás trabajando en un proyecto Maven para una aplicación web, no olvides incluir la dependencia JUnit en tu archivo pom.xml para comenzar con las pruebas:


                    <dependency>
                    <groupId>junit</groupId>
                    <artifactId>junit</artifactId>
                    <version>4.12</version>
                    <scope>test</scope>
                    </dependency>

Para aplicaciones web, es esencial probar las interacciones del servidor, los controladores, modelos y otras componentes del backend.

Las anotaciones en JUnit te ayudan a definir cómo se comportan tus pruebas y en qué momento se ejecutan ciertos métodos. Aquí te presento algunas de las más utilizadas:

  • @Test: Define un método como un caso de prueba.
  • @Before: Se ejecuta antes de cada prueba.
  • @After: Se ejecuta después de cada prueba.
  • @BeforeClass: Se ejecuta una vez antes de iniciar todos los métodos de prueba.
  • @AfterClass: Se ejecuta una vez después de finalizar todos los métodos de prueba.
  • @Ignore: Ignora el método como caso de prueba.

Estas anotaciones son esenciales para organizar y estructurar tus pruebas, asegurando que se ejecuten en el orden y momento adecuado.

Las aserciones son fundamentales en las pruebas unitarias, ya que nos permiten verificar que un método o función está retornando el resultado esperado. JUnit proporciona varios métodos para realizar estas comprobaciones en tus pruebas:

  • assertEquals: Comprueba si dos valores son iguales. Es útil para verificar que el resultado de una función coincide con el valor esperado.
  • assertTrue: Verifica que una condición sea verdadera. Puede ser utilizado para comprobar, por ejemplo, que un usuario esté autenticado.
  • assertFalse: Verifica que una condición sea falsa.
  • assertNotNull: Comprueba que un objeto no sea null. Útil para verificar, por ejemplo, que un objeto de usuario se haya recuperado correctamente de una base de datos.
  • assertNull: Verifica que un objeto sea null.
  • assertSame: Comprueba que dos referencias de objeto apunten al mismo objeto.
  • assertNotSame: Comprueba que dos referencias de objeto no apunten al mismo objeto.

Estas aserciones te ayudarán a asegurarte de que tu lógica de servidor y las interacciones con bases de datos, autenticación y otros procesos del backend estén funcionando correctamente.

En el desarrollo de aplicaciones web, es común enfrentar errores y excepciones, ya sean errores del servidor, problemas con solicitudes HTTP, errores de base de datos, entre otros. Estos errores pueden interrumpir la experiencia del usuario y causar problemas en la aplicación.

JUnit ofrece herramientas para probar estas excepciones de manera efectiva, permitiéndonos simular ciertos escenarios y verificar que nuestra aplicación maneja los errores adecuadamente. Una de las técnicas más comunes es el uso de la propiedad expected en la anotación @Test.


@Test(expected = MiExcepcionPersonalizada.class)
public void miMetodoDePrueba() {
    // código que se espera que lance MiExcepcionPersonalizada
}

Con el código anterior, esperamos que el método miMetodoDePrueba lance una excepción de tipo MiExcepcionPersonalizada. Si se lanza dicha excepción, la prueba pasará; de lo contrario, fallará.

Es esencial manejar y probar adecuadamente las excepciones en las aplicaciones web para garantizar una operación suave y proporcionar retroalimentación adecuada a los usuarios o tomar medidas correctivas en caso de errores.

Probar un Servlet implica simular un entorno que emule las solicitudes y respuestas HTTP. JUnit, junto con Mockito, es una herramienta popular para hacer esto. A continuación, un simple ejemplo:


import static org.mockito.Mockito.*;
import javax.servlet.http.*;
import org.junit.Test;

public class MiServletTest {

    @Test
    public void pruebaMiServlet() throws Exception {
        HttpServletRequest request = mock(HttpServletRequest.class);       
        HttpServletResponse response = mock(HttpServletResponse.class);    

        when(request.getParameter("miParametro")).thenReturn("valor");

        new MiServlet().doGet(request, response);

        verify(response).setContentType("text/html");
        // Más aserciones o verificaciones aquí
    }
}

Este código es una representación básica de cómo se pueden simular solicitudes y respuestas HTTP usando Mockito para probar un Servlet.

Las pruebas directas en páginas JSP son complicadas debido a su naturaleza integrada con HTML y Java. Sin embargo, se pueden probar los fragmentos de código Java dentro de JSP separadamente o usar herramientas de integración como Selenium para pruebas de interfaz de usuario. Aquí un enfoque con Selenium:


import org.openqa.selenium.WebDriver;
import org.openqa.selenium.WebElement;
import org.openqa.selenium.By;
import org.openqa.selenium.firefox.FirefoxDriver;
import org.junit.Test;

public class MiJSPPaginaTest {

    @Test
    public void pruebaTituloJSP() {
        WebDriver driver = new FirefoxDriver();
        driver.get("http://miweb.com/miJSP.jsp");

        WebElement title = driver.findElement(By.tagName("h1"));

        assertEquals("Mi Título Esperado", title.getText());

        driver.quit();
    }
}

Este código es un ejemplo básico de cómo usar Selenium para probar una página JSP, verificando el contenido de un elemento del título.
